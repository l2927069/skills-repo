# 需求文档 → 测试用例 (Req2TestCase)

读取 Word（.docx）格式的需求文档，解析需求内容，生成格式化的 Excel（.xlsx）测试用例文件。覆盖功能测试、边界测试、异常测试、业务规则测试等类型。

## 前置依赖

- **Node.js >= 18**（安装后首次使用需执行下方安装命令）

## 安装

Skill 安装后在工具目录执行依赖安装：

```bash
cd tools && npm install
```

> 💡 如需 Python 版本，可自行安装 `python-docx` 和 `openpyxl` 后使用，工具逻辑相同。

## 触发方式

在 Claude Code 中描述需求文档路径即可触发。例如：

> 读取 D:\需求文档.docx，生成测试用例
> 对这个需求文档生成测试用例: requirements.docx
> 帮我分析这份 Word 需求文档并生成测试用例

## 输出字段

Excel 包含以下 8 列（中文表头）：

| 列名 | 说明 | 示例 |
|------|------|------|
| 用例编号 | TC-{模块缩写}-{序号} | TC-LOGIN-001 |
| 所属模块 | 对应的功能模块 | 登录模块 |
| 用例名称 | 简明描述测试场景 | 验证正确用户名密码登录成功 |
| 优先级 | P0/P1/P2/P3 | P0 |
| 前置条件 | 测试前必须满足的条件 | 用户已注册，账号已激活 |
| 测试步骤 | 操作的详细步骤（分步编号） | 1. 打开登录页\n2. 输入用户名\n3. 点击登录 |
| 预期结果 | 期望的系统行为 | 登录成功，跳转首页 |
| 用例类型 | 功能测试/边界测试/异常测试/性能测试/安全测试 | 功能测试 |

## Excel 格式

- 蓝色表头（#4472C4 白字），冻结首行
- P0 用例编号红色加粗，P1 橙色标记
- 自动换行，自适应列宽
- 自动筛选器

## 工作流程

### Step 1: 提取需求文档文本

```bash
node tools/req2testcase.mjs extract "<文档路径.docx>"
```

输出为 JSON 结构，包含：
- `headings` — 文档标题层级（识别功能模块）
- `tables` — 文档中的表格数据
- `rawText` — 纯文本内容（供 AI 分析）
- `html` — HTML 结构文本

### Step 2: AI 分析需求 → 生成测试用例

基于提取的文本，分析以下内容：

1. **识别功能模块** — 根据文档标题和章节划分模块
2. **提取显式需求** — 文档中明确描述的功能点
3. **推断隐式需求** — 边界情况、异常流程、空值、超时等
4. **需求补全标记** — 歧义处标注「待确认」，缺失必要内容标注「补充建议」

对每个功能点，至少覆盖以下测试类型：

- **功能测试** — 正常流程、分支流程
- **边界测试** — 输入上限/下限、空值、特殊字符
- **异常测试** — 错误输入、网络中断、超时、权限不足
- **业务规则测试** — 条件组合、状态流转
- **安全测试** — 鉴权绕过、SQL注入、XSS（如适用）

### ⏸ Step 3: 导出草稿 → 用户评审

将 AI 生成的测试用例导出为 **草稿 Excel**，供用户打开查看和修改：

**通用 Excel 草稿：**
```bash
node tools/req2testcase.mjs generate testcases.json "<文件名>_待评审.xlsx"
```

**MeterSphere 格式草稿：**
```bash
node tools/req2testcase.mjs metersphere testcases.json "<文件名>_待评审.xlsx"
```

向用户呈现概要：

```
📋 测试用例生成完毕，共 X 条
   草稿已导出到桌面：xxx_待评审.xlsx

=== 按模块统计 ===
  xxx: X 条
  yyy: Y 条

=== 按优先级统计 ===
  P0: X 条 | P1: Y 条 | P2: Z 条

=== 按类型统计 ===
  功能测试: X 条 | 边界测试: Y 条 | 异常测试: Z 条
```

**等待用户操作：**
- 用户打开 Excel 审阅
- 用户回复：
  - "确认" → 进入 Step 4 导出最终版
  - "修改..." → 根据反馈调整用例，重新导出草稿
  - "补充..." → 补充遗漏场景

### Step 4: 导出最终文件

用户确认后，导出不带"待评审"标记的最终文件：

**通用 Excel：**
```bash
node tools/req2testcase.mjs generate testcases.json "<文件名>.xlsx"
```

**MeterSphere 格式：**
```bash
node tools/req2testcase.mjs metersphere testcases.json "<文件名>_MeterSphere.xlsx"
```
导入路径：测试跟踪 → 功能用例 → 导入 → Excel → 步骤描述

### 🔄 Step 5: 自动上传到 MeterSphere（可选）

无需手动导入，直接通过 API 上传：

**1. 准备配置文件** `ms-config.json`：
```json
{
  "url": "http://192.168.1.100:8081",
  "projectId": "你的项目ID",
  "auth": {
    "username": "admin",
    "password": "你的密码"
  },
  "batchSize": 10,
  "debug": false
}
```

**2. 执行上传：**
```bash
node tools/req2testcase.mjs upload testcases.json ms-config.json
```

上传过程自动：
- ✅ 登录获取 Token
- ✅ 按模块路径自动创建/匹配模块
- ✅ 批量并发上传（每批 10 条）
- ✅ 显示进度和结果

> ⚠️ 首次使用需确认 API 地址和端口。如果接口返回格式有差异，根据实际错误调整 request body 字段名即可。

## 优先级定义

| 级别 | 定义 |
|------|------|
| P0 | 核心功能，不通过则阻断发布 |
| P1 | 重要功能，建议修复后发布 |
| P2 | 次要功能，可延期修复 |
| P3 | 边缘场景 / 体验优化 |

## 注意事项

1. 需求歧义处标注为「⚠️ 待确认」
2. 需求缺失但必要的内容追加标注为「💡 补充建议」
3. 输出前检查用例是否完全覆盖了所有需求点
4. 文档中有图表/流程图时，优先识别文字描述，无法解析的标注为「📎 需人工确认」
5. **文件保存路径规则**：
   - 用户指定路径 → 保存到指定位置
   - 未指定路径但需求文档在桌面 → 保存到桌面
   - 未指定路径且需求文档在其他目录 → 保存到需求文档同目录
   - 草稿文件命名：`{需求文档名}_待评审.xlsx`
   - 最终文件命名：`{需求文档名}_MeterSphere.xlsx`
